# Copilot Instructions — Student Documents CRUD

Context for any AI assistance (GitHub Copilot, etc.) working on the **Student Documents**
feature in this repo. Read this before generating code for
`resources/views/registrar/student-documents.php` or anything that talks to
`app/controllers/registrar/StudentsDocumentController.php`.

## Files in this feature

| File | Role |
|---|---|
| `app/controllers/registrar/StudentsDocumentController.php` | MVC controller + bootstrap (handles POST for create/update/delete, GET AJAX for student search) |
| `app/models/registrar/StudentsDocumentModel.php` | Prepared-statement CRUD against `student_documents` table |
| `app/helpers/fileUpload.php` | `FileUpload::upload()` / `FileUpload::delete()` — moves files into `storage/<folder>/` |
| `app/services/StudentsService.php` | Has `searchStudents($keyword)` — the LIKE-search pattern to reuse |
| `resources/views/registrar/student-documents.php` | The page shell — currently only includes sidebar/topbar, **table + modals still need to be built here** |

The backend side of items below has already been fixed in this PR/branch. This document
exists so the **frontend** (the part Copilot doesn't have full context for from the backend
files alone) follows the same conventions.

## 1. Student picker must be search-based, NOT a `<select>` dropdown

Do not populate a `<select>` with every row from `students`. The table will keep growing,
so loading the full list on every page load doesn't scale.

Instead, mirror how `StudentsService::searchStudents($keyword)` already works elsewhere in
the app:
- Minimum 2 characters before searching.
- `LIKE '%keyword%'` against `lrn`, `first_name`, `middle_name`, `last_name`.
- Results ordered by `first_name`, `last_name`.

The controller already exposes this over AJAX:

```
GET student-documents.php?action=search_students&keyword=<text>
-> JSON array of student rows (id, lrn, first_name, middle_name, last_name, ...)
```

In the Add/Edit Document modal:
- Render a text input (e.g. `#student-search`) instead of a `<select name="student_id">`.
- Debounce keystrokes (~300ms), call the endpoint above, render matches in a dropdown
  list under the input.
- On selecting a match, show the student's full name + LRN in the input (read-only display)
  and store the actual `student_id` in a hidden field (`<input type="hidden" name="student_id">`)
  that gets submitted with the form.
- Block form submission if no student has actually been selected (i.e. the hidden
  `student_id` is empty), even if text is typed in the search box.
- When editing an existing document, pre-fill the search box with the current student's
  name/LRN and the hidden field with their id.

A small vanilla-JS typeahead is enough — no need to pull in Select2/Choices.js unless one
of those is already a dependency elsewhere in the app.

## 2. Status field

`student_documents.status` is an enum: `Pending`, `Submitted`, `Verified`, `Rejected`.

- The **default** status — both when the column has no value yet and when the Add
  Document form is first opened — is **`Submitted`**, not `Pending`. The controller
  bootstrap already defaults to `'Submitted'` when `$_POST['status']` is missing; make
  sure the `<select name="status">` in the modal has `Submitted` as the pre-selected
  `<option>` for the **create** form.
- The Edit form should pre-select whatever status the document currently has.

## 3. `fileUpload.php` path resolution (already fixed)

`storage/` lives at the **project root**, a sibling of `app/`, `database/`, `public/`,
`resources/`. Since `fileUpload.php` lives at `app/helpers/fileUpload.php`, the correct
base path is `dirname(__DIR__, 2)` (helpers → app → root), not `dirname(__DIR__, 3)`
(which escapes above the project root entirely). Both `upload()` and `delete()` were
fixed to use `dirname(__DIR__, 2)`. If you touch this file again, keep that in mind —
don't reintroduce the off-by-one.

## 4. Other bugs fixed while wiring up the CRUD

These were found while reading the existing controller/model and are now fixed —
mentioned here so new code stays consistent:

- The bootstrap was building the create/update payload with a `created_by` key, but
  `StudentsDocumentModel::create()`/`update()` bind to `$data['uploaded_by']` (matching
  the `student_documents.uploaded_by` column). Payload key is now `uploaded_by`.
- `delete()` called `$this->model->getById($id)`, which doesn't exist on
  `StudentsDocumentModel` — only `find($id)` is defined. Now uses `find($id)`, and fetches
  the row **before** deleting it (so `file_path` is still available to pass to
  `FileUpload::delete()`).
- `update()` was an empty stub. It's now implemented: it calls `find($id)` to keep the
  existing `file_path` when no new file is uploaded, then persists via
  `StudentsDocumentModel::update()`, logs the action, and redirects with a flash message
  — mirroring the `create()`/`delete()` pattern.

## Conventions to keep when writing new code here

- DB access goes through mysqli prepared statements (`bind_param`, never raw
  interpolation) — see `StudentsDocumentModel` / `StudentsService` for the pattern.
- Every mutating action (create/update/delete) logs through `AuditLogs::log()` and sets
  a flash message via `FlashMessage::setFlash()` before redirecting.
- AJAX endpoints live inside the same controller bootstrap, gated by `$_GET['action']`
  (see `search_students`), and `exit()` immediately after `echo json_encode(...)` — don't
  let execution fall through into the page-render path.
